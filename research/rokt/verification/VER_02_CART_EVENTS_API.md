## Cross-checks / caveats for a prototype

- **Path prefix:** Every Cart API path is `/v1/...`. The claim list dropped `/v1` on four of five paths and on `/confirmation/cancel`. Use `/v1/cart/reserve`, `/v1/cart/confirm`, `/v1/cart/release`, `/v1/confirmation/cancel`, `/v1/placements/any`.
- **`confirmation/cancel` is post-purchase**, distinct from `cart/release` (pre-confirmation reservation abandonment). Both use `itemReservationId`.
- **Dedup key drift between API versions** is the biggest trap: current Event & Audience API = `conversiontype`+`confirmationref`; legacy Event API also honors `transactionid`. Pin your dedup logic to whichever API version you actually target.
- **`emailsha256` vs `user_identities.other`:** the hashed-email *field name* differs between the legacy Event/Conversions API (`emailsha256`) and the current Event & Audience API (SHA-256 email carried under `user_identities`). DOUBLE-CHECK before hard-coding.
