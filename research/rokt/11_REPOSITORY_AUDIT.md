## Dreamship (product name in-repo: "AirLock")

- **Product thesis:** A "preflight" seller-operations layer between marketplaces (Shopify/Etsy/TikTok/Walmart) and Dreamship print-on-demand fulfillment. Ingests marketplace order payloads, normalizes to Dreamship's `ApiWriteOrder` shape, runs a deterministic validation pass, intercepts orders that would be rejected, and lets a seller fix → replay → recover them before the order locks at `accepted` (the immutable point of no return).
- **Tech stack:** Django 5.1 + DRF, Celery 5.4 + Redis, Postgres 16 (psycopg3, JSONB), Pillow, drf-spectacular. Two frontends: a Vite/React 18 + TanStack Query + Zustand operator console (`frontend/`) and a Next.js 16 + React Three Fiber/GSAP marketing "experience" (`experience/`). docker-compose (db/redis/backend/worker/frontend).
- **Strongest reusable BACKEND pattern:** A single authoritative **replay orchestrator with row-locking + terminal-state guard + double-submit protection at the irreversible boundary** — `backend\preflight\engine\replay.py`. Every replay kind (auto-retry, manual, dead-letter, approved-repair, first-submit, reconcile) funnels through one `replay_order` path that: locks the order row with `select_for_update` and holds it across guard→revalidate→submit; returns an idempotent no-op if the order already reached SUBMITTED (can't walk backward); re-checks idempotency immediately before the supplier boundary in `supplier._submit_locked`; and appends ReplayJob attempts (never overwrites) under a stable `correlation_id`. Reinforced by a **DB-enforced partial-unique idempotency constraint** (`backend\preflight\models.py` `uniq_live_idempotency_key`, partial on `is_simulation=False AND idempotency_key != ''`) with a data-migration that blanks live keys *before* adding the constraint (`backend\preflight\migrations\0007_*.py`); an **HTTP `Idempotency-Key` decorator** storing a body-hash and returning 409 on key reuse with a different body (`backend\preflight\idempotency.py`); and an immutable append-only `AuditEvent` spine stamped with correlation IDs via context-var middleware (`backend\preflight\engine\audit.py`, `backend\preflight\middleware.py`).
- **Strongest reusable FRONTEND pattern:** A single typed API client that centralizes every endpoint as a named function returning unwrapped typed data (`frontend\src\lib\api.ts`) — one axios instance, an `endpoints` object mapping routes to typed `Promise<T>`, consumed via TanStack Query; UI-only state (active order, filters) in a tiny Zustand store (`frontend\src\store\useAppStore.ts`). Clean server-state / client-state separation.
- **Weak/generic patterns to AVOID:** `ALLOWED_HOSTS=*`, `DJANGO_SECRET_KEY="dev-insecure-change-me"`, `AUTH_PASSWORD_VALIDATORS=[]`, and no real API auth (throttle-only `600/min`); hardcoded compose DB creds (`dreamship/dreamship`) with `runserver` + `makemigrations` at container start; the Next.js `experience/` 3D marketing layer is presentation, not reusable logic; `PREFLIGHT_STAGE_DELAY_SECONDS` artificial `time.sleep` is demo theater.
- **Reusable components / infra / test-arch / seeded storytelling:** Standout test suite — 40+ focused pytest modules under `backend\preflight\tests\` incl. `test_idempotency.py`, `test_hostile_input.py`, `test_input_hardening.py`, `test_money_precision.py`, `test_invariants.py`, `test_rule_index_n1.py` (N+1 guard), `test_replay_orchestration.py`, `test_tenancy.py`. Seeded storytelling: `backend\preflight\seed\orders.py` builds authentic per-platform raw payloads for **six signature failure modes** and runs them through the *real* engine (ingest→normalize→validate) so seed data is pipeline-produced, orchestrated by `management\commands\seed_demo.py`. Reusable UI primitives: `ScoreRing.tsx`, `Timeline.tsx`, `JsonPane.tsx` (Monaco raw→normalized split), `Drawer.tsx`. Correlation-ID middleware + log filter is drop-in infra.
- **Technical debt:** No API auth layer; insecure default secret/hosts; compose runs the dev server and auto-`makemigrations`; artificial pipeline delays; two divergent frontends double the maintenance surface; marketplace OAuth unimplemented (payloads are disclosed mocks).

## Efficast (product name in-repo: "Verified Recovery Agent" / VRA)

- **Product thesis:** An autonomous agent for discrete manufacturing that begins *after* a maintenance intervention: drafts a machine-readable **Recovery Contract**, blocks until required human evidence is fresh+valid, observes the real post-intervention cycle trajectory, then closes / conditionally accepts / reopens / escalates — refusing to let "work order complete" masquerade as a recovered factory. The LLM proposes but never judges recovery or actuates.
- **Tech stack:** FastAPI + SQLModel + SQLite (Postgres/pgvector in prod), Pydantic v2, numpy, httpx; optional LangGraph/langchain-core (a deterministic provider carries the full demo with no API key); MCP server (`app/mcp_server.py`). Frontend: Next.js 14 + React 18, TanStack Query, Radix UI, framer-motion, Zod, cmdk. docker-compose (pgvector/pg16 + backend + frontend).
- **Strongest reusable BACKEND pattern:** A **tamper-evident, hash-chained + HMAC-signed audit log paired with a transactional outbox** — `backend\app\workflow\audit.py`. Each `AuditEvent` carries a monotonic per-correlation `seq`, a deterministic content `entry_hash` chained to the previous entry's hash (the hashed payload includes attribution: incident/contract/plant/model/prompt version), plus an optional keyed `entry_hmac` (HMAC-SHA256, key indirected through `current_hmac_key()` for KMS/vault sourcing). `verify_audit_chain()` recomputes the chain and detects tampering at the first broken seq, distinguishing "hash chain recomputed by a DB attacker" from "signature forged." Outbox events are written in the *same transaction* as the state change and drained exactly-once by `drain_outbox()` with retry classification → dead-letter after `MAX_OUTBOX_ATTEMPTS`. Complemented by a **durable state machine** with a legal-transition table + role guards + optimistic-lock version and a single `transition()` entry point (`backend\app\workflow\state_machine.py`); a **per-tool circuit breaker** closed→open→half_open (`backend\app\gateway\circuit.py`); and an **idempotency ledger** for once-only write effects (`backend\app\gateway\idempotency.py`).
- **Strongest reusable FRONTEND pattern:** A typed API layer with a first-class `ApiError` carrying `status`/`detail`/`code`/`stage` (`frontend\lib\api.ts`), paired with a **complete, accessible state-component kit** in `frontend\components\forge\states.tsx` — `Skeleton`, `LoadingState` (`role="status"`/`aria-live`), and generative "awaiting signal" empty-state illustrations honoring `prefers-reduced-motion`. RBAC threaded through the client via `setApiUser`/`lib\role.tsx` (X-VRA-User header synced to the active principal), plus an `announce.ts` a11y live-region helper. A reusable "every screen has a designed loading/empty/error/locked/offline state" system.
- **Weak/generic patterns to AVOID:** Header-based demo identity (`X-VRA-User` → seeded user) with a demo default principal `s.vega` — a documented seam that must never ship as real auth; `VRA_DEMO_MODE=1` in compose; hardcoded compose creds (`vra/vra`). ~60 service modules under `app\services\` risk over-fragmentation (reuse the "one service per bounded concern" idea, not the sheer count).
- **Reusable components / infra / test-arch / seeded storytelling:** Marquee reusable pattern is the **architecture fitness-function suite** — `backend\tests\test_architecture.py` parses the `app/` import graph via `ast` and fails the build if hexagonal boundaries break (domain imports no outer layer; agent/reasoning never import the gateway; tool registry only reachable through the gateway; **no machine-control function exists anywhere in source**), documented in `docs\adr\0004-*`. Full ADR set (`docs\adr\0001-0004`). Bounded **agent graph** (`backend\app\agent\graph.py`): a Reflexion-shaped Plan-Executor (perceive→retrieve→hypothesize→draft→self_critique→decide, then observe/reflect) recording every step as an inspectable `AgentReasoningTrace`, LangGraph-liftable without touching node logic. Seeded storytelling: `backend\app\seed\northstar.py` — the deterministic "Northstar Packaging Plant" scenario seeds only the substrate *up to the moment the agent takes over*; relapse/verification run live. Security seam: `IdentityProvider` port with an unwired `OidcIdentityProvider` shape and an explicit role→permission least-privilege matrix (`backend\app\security.py`) where the agent principal can propose/monitor but never approve/release/close.
- **Technical debt:** Demo-mode default principal must be gated off in prod; hardcoded compose creds; ~60 narrow service modules + 60+ markdown docs imply high doc/code coupling; the optional LangGraph path is under-exercised (the deterministic provider carries the demo); SQLite-default vs pgvector-prod split diverges the vector/RAG path.

## Episode Companion Agent

- **Product thesis:** A single-episode RAG microservice that turns one "AI Research Daily" episode into a mini-agent, answering in three fixed personas (Plain English / Founder Takeaway / Engineer Angle) plus learning modes (quiz-me, self-explain, cross-episode timeline). An upstream pipeline auto-generates episodes from arXiv.
- **Tech stack:** FastAPI + Uvicorn; LangChain over Ollama (`qwen2.5:7b-instruct`) with OpenAI/Gemini stubbed behind a factory; ChromaDB + BM25 (`rank_bm25`); SQLAlchemy + SQLite; Alembic + APScheduler declared; `slowapi`/`tenacity` present; vanilla HTML/CSS/JS frontend in `static/`. No containerization.
- **Strongest reusable BACKEND pattern:** A layered **RAG grounding / anti-hallucination pipeline** in `agent.py` stacking four independent defenses: (1) hybrid retrieval with **Reciprocal Rank Fusion** of vector + BM25 (`_reciprocal_rank_fusion`, `_retrieve_gpk`); (2) a deterministic **allow-list guardrail** hard-blocking named entities not present in episode content, returning a fixed `INSUFFICIENT_MSG`; (3) an **LLM-as-critic retry loop** (critic judges grounding, one retry with wider `k`, falls back if still ungrounded); (4) deterministic structural validators (`_check_structure`, `_validate_answer`). Every response carries a trace envelope (`trace_id`, per-stage latency, token estimates, quality_checks) — a reusable observability layer on its own. Also strong: race-safe **repository + get-or-create** with deferred-commit transactions (`repositories\conversation_repository.py`, `conversation_manager.py`).
- **Strongest reusable FRONTEND pattern:** Token-driven design system in `static\css\design-system.css` (CSS custom properties: color scales, gradients, glassmorphism, dark surfaces) consumed by `components.css` — a clean framework-free theming layer. Per-mode `suggested_followups` arrays driving guided-prompt chips is a reusable conversational-UX pattern.
- **Weak/generic patterns to AVOID:** Massive `if/elif` keyword classifier in `behavior.py` (brittle substring matching, won't internationalize); hardcoded episode-specific `special_terms` guardrail list (will silently rot); `CORSMiddleware` `allow_origins=["*"]` **plus** `allow_credentials=True` in `main.py` (invalid/insecure combo); default API key literal `"my-secret-antigravity-password"`; manual `asyncio.get_event_loop()`/`run_until_complete` inside sync handlers. Poor repo hygiene: ~60 loose `test_*`/`debug_*`/`verify_*` scripts at root, committed `.env`, committed `chroma_db/` and `.db` binaries, empty `alembic/versions/`.
- **Reusable components / infra / test-arch / seeded storytelling:** Pluggable **LLM client factory** (`llm_client.py`, ABC + backend switch). Proper pytest suite under `tests/` (`test_guardrail.py`, `test_rag_quality.py`, `test_repositories.py`, `test_orchestrator.py`) asserting both block and allow paths. Standalone cron-style **data-retention janitor** `jobs\cleanup_conversations.py` (callable via `/cleanup`). Seeded storytelling: `data\sample_episode.txt` + `data\daily_report_11_18.txt` give a real narrative episode ("Kandinsky 5.0") that the guardrail tests are written against.
- **Technical debt:** OpenAI/Gemini backends raise `NotImplementedError`; README documents endpoints/flags that don't fully exist; two parallel entry points (`main.py`, `backend/server.py`); Alembic configured but unused (schema via `init_db()`/`create_tables.py`); debug `print()` in hot paths; many churny status markdowns (`PRODUCTION_READY.md`, `STATUS.md`, `TEST_FAILURES.md`).

## fanflow-ai-complete (FanFlow AI)

- **Product thesis:** A post-purchase "event-day guidance" layer that converts a ticket confirmation into a personalized arrival plan (recommended gate, leave-by time, support points, confidence) with live staff/fan condition signals. Load-bearing thesis: **rules decide, AI only explains** — one real fan (Maria), one venue, one complete journey.
- **Tech stack:** Next.js 14 App Router + TypeScript + Tailwind; Zod; Framer Motion + Leaflet. LLM via raw `fetch` to Groq (`llama-3.3-70b`) → Gemini (`1.5-flash`) → template cascade, no SDKs. State today is `localStorage` + `storage` events behind a `FanflowStore` interface; documented Supabase Postgres + Realtime swap (`supabase/schema.sql`, 4 tables). CI (GitHub Actions: typecheck + 133 vitest tests + lint); Vercel-ready.
- **Strongest reusable BACKEND pattern:** The **deterministic, explainable rule engine** in `lib\seed.ts` (`deriveArrivalPlan` / `scoreGateBreakdown`). Every gate is scored into seven transparent components (section-proximity tiers, accessibility/family/sensory match, wait penalty, staff signal, fan signal), returned as a `GateScoreBreakdown[]` so the UI renders a "Why this gate?" panel — no black box. Encodes a real trust model: staff signals weighted 3×, fan signals only counted once a ≥3 majority threshold is met, plus a derived confidence percent with conflict/staleness reasons. Equally reusable: the **anti-hallucination explain boundary** (`lib\explain\arrivalExplanation.ts` + `app\api\explain-arrival-plan\route.ts`) — Zod-validated request; provider cascade with per-call `AbortController` timeout **and** an outer `withTimeout` (defense in depth); regex banned-phrase `sanitizeExplanation` (450-char cap at sentence boundary, enforced safety note); and `mentionsWrongGate()` that discards any LLM output naming a gate other than the rule-chosen one (handles spelled-out numbers). The **SSE Live Wire** (`lib\server\signalHub.ts` + `app\api\signals\stream\route.ts`) adds server sequence numbers, a ring buffer, **idempotent publish by client id**, `?since=<seq>` catch-up replay (at-least-once + client dedupe), heartbeats, and unsubscribe-on-abort.
- **Strongest reusable FRONTEND pattern:** **Store-interface-driven reactive projection.** The `gameDaySim` singleton (`lib\simulation\gameDaySim.ts`) steps a scripted event timeline and publishes real `LiveSignal`s into the same store every surface subscribes to, so the whole app re-renders with zero per-page wiring. Combined with the pluggable `FanflowStore` (`lib\data\`, local vs supabase by env) and pure derived-intelligence modules (`lib\intelligence.ts`, `lib\nextMove.ts`, `lib\impact.ts`) feeding presentational cards (`components/hub/`, `SourceChip.tsx`, `ConfidenceChip.tsx`). SSR-safe (locale-independent `fmtTime` avoids hydration mismatch).
- **Weak/generic patterns to AVOID:** In-memory module state for the rate limiter (`lib\rateLimit.ts`) and signal hub is single-instance only (honest about it, but not multi-instance safe). The `staff-overrides-fan-noise` scenario admits the weighting math produces a tie resolved only by proximity tiebreak — the 3×-vs-threshold trust model is finely balanced and could invert with small tuning. ~60 `.agents/skills/` design-tooling files vendored into the repo bloat it. `localStorage` session isolation is spoofable, demo-only.
- **Reusable components / infra / test-arch / seeded storytelling:** Excellent **scenario-fixture test architecture** — `lib\scenarios\scenarios.ts` defines 7 personas as `(prefs, signals, expectedBehavior)` bundles that both drive `tests/rule-engine.test.ts` and power a live `/debug` toolbar (same fixtures, test + demo). 133 vitest tests across 10 files cover pure logic. Reusable infra: standardized error-envelope + `x-request-id` propagation, the `FanflowStore` swap seam, provider abstraction `lib/providers/`. Strong seeded storytelling: `lib\seed.ts` is a complete narrative world (FIFA WC 2026 Final, MetLife, Maria + 6-year-old, Section 117, three characterful gates) and `gameDaySim`'s `SCRIPT` is an authored dramatic arc (Gate 3 heats up, Gate 7 emerges as the calm alternative).
- **Technical debt:** `supabaseFanflowStore` is a stub; the entire Supabase/Realtime production path is documented but unimplemented (deliberate, well-labeled seam); two intentionally non-shared prototype layers (`/demo/fanflow-vision` state machine vs the real Hub) duplicate journey concepts; no E2E/UI tests; single hardcoded event (`wc2026-final`); rate limiter + signal hub need externalizing (Redis/Upstash) before real deployment. Debt is unusually well-documented (README "Honest limitations" table).

## Get Towed

- **Product thesis:** A consumer web app for people whose vehicles have been towed — look up an impounded vehicle by plate, view the fine, file a "plead query" (dispute), pay via Stripe, retrieve a receipt. A Flask admin panel manages towing companies, vehicles, payments, retrievals.
- **Tech stack:** Flask + SQLAlchemy + Flask-Migrate (Alembic) + Flask-JWT-Extended + Flask-Admin, SQLite, Stripe, Faker. Frontend Create-React-App (React 18, react-router-dom, Bootstrap). No Docker/deploy config.
- **Strongest reusable BACKEND pattern:** The **auto-generated CRUD admin panel via Flask-Admin ModelView subclasses** (`server\views.py` + registration in `server\app.py`). Each model gets a declarative `ModelView` with `column_list`, `column_labels`, relationship-aware `column_filters` (`'user.username'`, `'vehicle.plate_number'`) — a full back-office with near-zero code. The Werkzeug `create_password`/`check_password` pattern on `User` (`server\models.py`) is a clean reusable mixin idea. Beyond that the server is thin CRUD.
- **Strongest reusable FRONTEND pattern:** Standard `BrowserRouter`/`Routes` with a persistent `NavBar` outside `Routes` (`client\client\src\App.js`) — textbook but generic; no state-management or data-fetching abstraction to lift.

Duplicated near-identical routes (`/addvehicle` vs `/addvehiclecompany`); manual per-field `data.get()` with no serializer; hardcoded SQLite URI; `debug=True`.
- **Reusable components / infra / test-arch / seeded storytelling:** **Faker-based seeding** (`server\seed.py`) is the reusable asset (5 realistic rows/table). Test architecture effectively absent (only CRA default `App.test.js`). One autogenerated Alembic revision — no real migration history.
- **Technical debt:** Missing auth guards on all endpoints; client-trusted payment amounts; type-mismatched columns; `node_modules` committed; nested `client\client`; no `requirements.txt`; no error handling on write routes; competing schema sources (`db.create_all()` + migrations). **Overall: mostly an anti-pattern reference — do not copy.**

## NexusWatch MVP

- **Product thesis:** A sales-tax **economic-nexus monitoring** tool for finance teams ("Sara"): ingest invoices (manual, pasted text, PDF/OCR), categorize line items, compute per-state taxable totals against configured thresholds, flag risky invoices into a human review queue, export audit-ready CSVs — explicitly **decision support, not legal/tax advice**.
- **Tech stack:** Next.js 15 (App Router, React 19, TS) + Tailwind; Supabase (Postgres + Storage + RLS); Zod; TanStack Table + Recharts + lucide-react; OCR via `pdfjs-dist` + `@napi-rs/canvas` + `tesseract.js`. No test runner (lint/typecheck only). Vercel deploy.
- **Strongest reusable BACKEND pattern:** Two standouts. (1) **Layered OCR/text-extraction pipeline with graceful degradation** — `app\api\uploads\pdf\route.ts` tries embedded PDF text (geometry-aware line reconstruction `rebuildPdfTextLines`), falls back to a latin1 literal-string scrape, then rasterize-and-Tesseract (`lib\pdfOcr.ts`), producing a status ladder (`pdf_text` → `ocr_needs_review` → `manual_review_required`) with confidence scores; field parser `lib\invoiceTextParser.ts` emits `fieldConfidence`/`missingFields`/`warnings`. (2) **Deterministic rules engine + human-in-the-loop review state machine + audit trail** — `lib\nexus.ts` (pure threshold math: 75/90/100% bands, `previewInvoiceImpact`, `buildStateSummaries`) cleanly separated from persistence; `lib\supabaseWrites.ts` derives flags (`buildWritableFlags`), computes review/operational/risk status, and writes a structured `audit_logs` row with `actor` + `metadata` on every mutation. Also **schema-drift-tolerant writes** — detect Postgres `42703` (missing column) and retry with a reduced payload (`isMissingReviewColumnError`).
- **Strongest reusable FRONTEND pattern:** The **live impact-preview intake form** (`app\upload\page.tsx`) — as fields change it runs the same `previewInvoiceImpact`/`buildStateSummaries` domain logic client-side to show projected threshold movement, `projectedWarnings`, and a detected-fields preview (`ExtractedFieldsPreview`) with confidence pills before persisting. Sharing the pure `lib/nexus.ts` engine between client preview and server write is the reusable idea. A `demo → live` cookie toggle (`nexuswatch_data_mode`) is a clever data switch.
- **Weak/generic patterns to AVOID:** **Permissive RLS** — `supabase\nexuswatch_write_policies.sql` grants `anon` insert/update `with check (true)` on all tables (demo-only, a prod trap). **Hardcoded single-tenant identity** — `companyId` falls back to a demo constant; `actor` hardcoded `"Sara Demo User"`. **Regex PDF/invoice parsing is brittle** (mitigated by forcing manual review, but not durable). **In-memory module cache** (`lib\supabaseData.ts`, 5s TTL) is per-serverless-instance. Duplicated `Premium*` vs plain component variants suggest incomplete migration / dead code.
- **Reusable components / infra / test-arch / seeded storytelling:** **Design system** — HSL CSS-variable token theme in `tailwind.config.ts` + shared primitives (`StatusBadge`, `EmptyState`, `Toast`, `CsvExportButton`) and layout (`AppSidebar`, `TopBar`, `PageHeader`). **CSV export subsystem** — `lib\csv.ts` + `app\api\exports\route.ts` produce five typed export shapes with deterministic sanitized filenames. **Seeded storytelling** (model-grade) — `supabase\nexuswatch_mvp_seed.sql` + `lib\demoData.ts` seed a coherent narrative (company "Sara", 24 invoices, 5 state rules, 13 flags); `HANDOFF.md` documents the exact verified demo path and ordered SQL migration sequence. Test architecture is the gap — no unit tests despite highly testable pure modules.
- **Technical debt:** `node_modules` committed; demo-permissive RLS + hardcoded tenant/actor must be replaced; schema managed by a stack of hand-run idempotent `alter table ... add column if not exists` files with app-side fallback compensating — flexible but fragile; duplicated Premium/plain pairs; regex-parser maintenance burden; in-memory cache unsuited to serverless.

## ShelfTrace Control Plane

- **Product thesis:** A reliability control plane that verifies AI-approved retail price changes actually landed correctly across every channel (POS, ESL/shelf-label, ecommerce) before expanding a rollout zone-wide — catching checkout mismatches, timeouts, and perishable-markdown deadline risks before shoppers are overcharged. Deliberately a demo/simulation harness (simulated channel adapters, seeded pricing) showcasing execution-safety engineering.
- **Tech stack:** Python, FastAPI, SQLAlchemy 2.0 (typed `Mapped[]`), Alembic (13 migrations), Postgres (prod) + SQLite (tests), Redis, a standalone outbox worker (`app/worker.py`), optional OpenTelemetry, slowapi, MCP server. Frontend: Next.js App Router, TS, Tailwind, framer-motion, custom polling hook. `docker-compose.yml`, `render.yaml`, `verify.sh` smoke script.
- **Strongest reusable BACKEND pattern:** The **transactional outbox + worker with exponential backoff, dead-letter, and Postgres row-locking**. `backend/app/services/orchestrator.py` — `process_outbox_once()` selects `PENDING/RETRYING` events respecting `next_attempt_at`, and on Postgres adds `.with_for_update(skip_locked=True)` so inline drain and standalone worker never double-process; `_next_attempt_delay()` does capped exponential backoff *with jitter*; on exhausting `outbox_max_attempts` it transitions to `DEAD_LETTER` and calls `dead_letter_alert()` (`backend/app/services/dead_letter.py`, structured ERROR log + best-effort non-blocking Slack webhook). Paired with: a **deterministic reconciliation state machine** (`backend/app/services/reconciliation.py`, pure `decide_action()`: mismatch→BLOCKED, timeout→RETRY, all→ELIGIBLE; single enforcement point `_is_held_against_reconcile()` guarantees an integrity hold can never be lifted by channels merely agreeing on the wrong number); **idempotency keys** on scrape/outbox/import (`Idempotency-Key` header; unique `idempotency_key` columns; deterministic sha256 catalog loader); an **immutable append-only audit trail** with caller-supplied strictly-increasing timestamps for causal ordering (`backend/app/services/audit.py`); and **row-locked recovery** (`backend/app/services/recovery.py`, `with_for_update()`).
- **Strongest reusable FRONTEND pattern:** `frontend/lib/useLive.ts` — a polling/fetch hook that stamps every request with a monotonic `epoch` and commits only the latest epoch's result, defeating both the unmounted-setState warning and the slow-late-response-overwrites-fresh staleness bug. Dependency-free, reusable in any poll-driven dashboard (used by `frontend/app/(app)/operations/page.tsx` at 2s).
- **Weak/generic patterns to AVOID:** The reconciliation "engine" is driven entirely by pre-seeded `ConnectorBehaviorProfile` rows — no real channel integration (simulation replaying configured outcomes). Some routers embed reconciliation/incident-creation logic inline in the handler (a `GET /operations` that mutates + commits during a read) — mixing read endpoints with write side effects is not to copy. A large `/vision/*` marketing surface is presentation, not engineering.
- **Reusable components / infra / test-arch / seeded storytelling:** Test architecture is the real asset — ~65 test files incl. `test_concurrency_pg.py` (real Postgres row-lock/SKIP-LOCKED concurrency), `test_outbox_backoff.py`, `test_reconciliation.py`, `test_deadline_recovery.py`, plus executable "prove" scripts (`tests/prove_sequence.py`, `prove_certification.py`) driving the live API end-to-end, wired into `verify.sh` (spins up compose, waits on `/health`, runs pytest, proves the sequence, resets demo state). ## 100 Miles of Summer

- **Product thesis:** A ritual-based summer fitness challenge (move 100 miles, May–Aug) rewarding consistency and resilience over intensity, with fitness-provider sync (Strava/Garmin/Apple), crews, achievement "micro-wins," and compassionate support flows. Substantial parts are an interactive frontend demo with a partially-stubbed backend.
- **Tech stack:** Next.js 16 (App Router, React 19), TS, Tailwind v4, Radix + shadcn, framer-motion, zod, canvas-confetti, html-to-image. Data — a **dual-driver** setup: `better-sqlite3` (raw SQL, primary/local) plus Prisma 7 targeting Postgres/Supabase as an async "cloud sync" mirror. Tests — Vitest + Testing Library. SSE for live sync. Vercel serverless.
- **Strongest reusable BACKEND pattern:** **Provider-scoped idempotent ingestion via composite natural key + append-only correction ledger.** In `src/lib/services/activityService.ts`, `insertActivities()` uses `id = provider + ':' + external_activity_id` with `INSERT ... ON CONFLICT(provider, external_activity_id) DO UPDATE` inside a `better-sqlite3` transaction, counting `added` vs `dupes` and catching `SQLITE_CONSTRAINT` — clean dedup for at-least-once webhook/poll delivery (unit-tested in `src/test/sync.test.ts`). Corrections are **never destructive**: `correctActivity()` appends an additive `delta_miles` row to `activity_corrections` ("Immutable Audit Trail"), then `recalculateAggregates()` re-derives totals from an `activity_effective_miles` view (base + corrections). Plus an `ops_audit_log` table (admin_id/action/target_id/meta_json) written on every ops action (`src/lib/services/opsService.ts`). `providerSyncService.ts` adds a token-lifecycle state machine (revoked/expired→refresh→incident-on-failure) with simulator flags (rate_limit/delay/duplicates/outage) for fault injection.
- **Strongest reusable FRONTEND pattern:** The 8-step guided demo state machine under `src/components/demo/` (`DemoContext.tsx` + `OnboardingStep` … `SummerStoryStep`) driven by SSE from `src/app/api/demo/sync/stream/route.ts` for live sync progress with optimistic UI, plus a shareable recap card exported to PNG via html-to-image. A clean self-contained onboarding-narrative pattern on a shadcn/Radix design system.
- **Weak/generic patterns to AVOID:** The **"Holographic Database" adapter (`src/lib/db.ts`)** is the anti-pattern to avoid: in production/Vercel it silently swaps the real DB for an in-memory mock that pattern-matches on lowercased SQL substrings and returns canned data, with `transaction: (fn) => fn` (no atomicity) and `run: () => ({ changes: 1 })` (fake success) — the deployed app shows fabricated data and writes silently no-op. Also avoid the **schema/driver split** (Prisma declares postgresql while the app's primary path is raw-SQL SQLite, Prisma as fire-and-forget mirror that only logs on failure — divergence risk, no reconciliation). Many documented APIs are `❌ TODO` stubs; auth is a demo cookie. Some tests assert trivialities (`sync.test.ts` asserts string interpolation, not the dedup path).
- **Reusable components / infra / test-arch / seeded storytelling:** Reusable UI kit in `src/components/ui` (shadcn + Radix, `components.json`, cva + tailwind-merge `cn()`). The **micro-wins/achievement engine** design (Consistency Ace; The Comeback after a 10-day gap; Rhythm Maker time-of-day detection — `FEATURE_DOCUMENTATION.md` Feature 6) is a reusable behavioral-recognition spec. Strong seeded storytelling in demo/ops mock data: a "quiet list" user (`user_quiet`, "To feel alive", 20 days silent) surfaced for compassionate nudging; seeded incidents ("Rate limit approaching for Strava API"); resilience/comeback narrative baked into achievements. Privacy is first-class (`privacyService.ts`, `/api/demo/user/export` + `/delete`, `TrustCenter.tsx`). Focused service tests (`activityService.test.ts`, `providerSyncService.test.ts`, `tokenLifecycle.test.ts`).
- **Technical debt:** Production build runs entirely on mocked data (holographic adapter; real SQLite path commented out "for Vercel build stability") — deployed behavior diverges fundamentally from local; committed DB artifacts + `build.log` + `errors.txt`; dual source-of-truth invites drift; large gap between "verified working" framing and many `❌ TODO` routes; `ActivityCorrection` carries legacy dual-shape fields indicating an unfinished migration.
